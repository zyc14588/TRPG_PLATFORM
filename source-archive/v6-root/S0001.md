# AGENTS.md — COC AI TRPG Codex 持久化施工指令

本文件用于 Codex 的仓库级持久化指导。Codex 执行任何编码、测试、重构、修复或评审任务前都必须读取本文件，并结合 `docs/codex/00-index/codex-persistent-context.md` 与相关 per-file prompt 执行。

## 不可突破施工原则

- Authority Contract 不可变；变更只能 Fork 或新建 authority_contract_version。
- HUMAN_KP / AI_KP 在 Campaign 级互斥，任何 handler 都不得在同一 campaign 同时接受两种 KP authority。
- AI 不直接写正式状态；AI 只能提出 Proposal / ToolCall / DraftDecision。
- 正式状态必须走 Command -> Workflow -> Decision -> Event Store -> Projection。
- Event Store 是正史，Projection / Cache / RAG Index 均为可重建读模型。
- Visibility Label 与 Fact Provenance 必须跨 API、Event、Agent、RAG、Export、Log、Metric 传播。
- Tool Grant、Policy Gate、OpenFGA、OPA、Audit Log 不得被 Agent、插件或外部 Provider 绕过。
- 所有写命令必须具备 idempotency_key、expected_version、actor、correlation_id、causation_id。

## Codex 工作方式

1. 先定位任务对应的 `codex-prompts/<category>/P####.md/<prompt>.md`，再读模块级 `docs/codex/<NN-category>/codex-module-code-prompt.md` 与 `codex-module-test-prompt.md`。
2. 修改代码前先给出最小执行计划，列出将改动的 crate、migration、API/Event schema、测试文件。
3. 不要引入 `ModuleService`、`ModuleCommand`、`ModuleError` 或“核心接口占位”等模板残留；所有公开类型必须使用领域专名。
4. 不要把 `serde_json::Value` 当作最终领域模型；只允许在 provider adapter、unknown external payload、审计附加信息中受控使用，并必须有 schema boundary。
5. 不要绕过 SQLx transaction、Event Store append、outbox、projection rebuild、audit log 或 policy gate。
6. 每次新增写命令都必须同时新增正向测试、幂等冲突测试、版本冲突测试、权限拒绝测试和 visibility 泄露负例。
7. 每次新增 API/Event/WebSocket/NATS 契约都必须同步更新 schema、contract test、trace map。
8. 运行 `cargo fmt --all -- --check`、`cargo clippy --workspace --all-targets --all-features -- -D warnings`、`cargo test --workspace --all-features`；受环境限制无法运行时，必须说明原因和替代验证。

## 施工目录约定

- `docs/codex/00-index/`：Codex 总控、执行图、持久化上下文、源到 prompt 映射。
- `docs/codex/01-foundation/` 到 `docs/codex/12-extension-sdk/`：模块级编码、测试、评审提示词。
- `codex-prompts/`：每个输入 Markdown 文件生成的一份 Codex 施工提示词。
- `docs/codex/99-appendix/`：Prompt 模板、官方 Codex 参考、测试命令清单、遗留文档处置说明。

## 输出标准

每个 Codex 任务完成后必须输出：变更文件清单、测试命令与结果、schema/migration/event/API 变更、风险与 TODO、关联 Prompt ID。没有测试或没有验收证据的实现不得合并。

## Strict Fixed：crate / module / task_type 权威

- per-file prompt 的 `建议 crate` 必须与模块级默认 crate 一致，禁止使用旧别名或表格污染字段。
- Rust module 名称不得携带 hash 后缀；hash 只保留在 Prompt ID、源 SHA、Prompt 文件名和追踪矩阵中。
- `docs-governance` 与 `traceability-maintenance` 任务默认只维护 Markdown、索引、矩阵、报告、验证清单或文档治理工具，不得创建业务域写模型、业务 API handler、业务 NATS subject 或正式状态写入路径。


## Strict Module Layout：Rust 文件布局权威

- Codex 执行任何 Rust module 施工时，默认使用 flat module：flat src module file。
- 禁止同一任务、同一 PR 或同一 prompt 同时创建 flat 文件与目录式 `mod.rs` 文件；两种布局必须二选一。
- 当模块复杂到需要子模块时，允许改用目录式布局，但必须删除对应 flat 文件、同步更新 `lib.rs`、测试、文档追踪和 review 说明。
- per-file prompt 的“建议落地文件”是文件布局的唯一执行入口；若发现冲突，以 `STRICT_CODEX_SEMANTIC_VALIDATION.md` 的校验规则为准并先修复 prompt。

## 第四次严格修复：输出路径一致性

- 对 `product-code`、`test-harness`、`ops-runbook` prompt，Rust 输出文件必须由 `建议 crate` 与 `建议 Rust module` 推导：`crates/<crate>/src/<module_tail>.rs` 与对应 contract test。
- 不得从源文档路径、旧 V5 中间文件名、截断路径或 hash 片段生成 `docs_implementation_*`、`source_breakdown_*`、`generated_from_source_*` 伪 Rust 文件。
- 同一 concrete Rust 输出路径只能有一个 `primary-implementation` prompt；其他同 module prompt 必须是 `supplemental-requirement`，不得重复创建同一 `.rs` 文件。
- `docs-governance` 与 `traceability-maintenance` prompt 默认只维护 Markdown、索引、矩阵、报告、校验清单和追踪材料。

## 第五次严格修复：Supplemental 语义与稳定 module 命名

- `primary-implementation` 是唯一可以声明 concrete Rust src/test 输出文件的角色。
- `supplemental-requirement` 只生成 `docs/codex/90-traceability/supplemental-requirements/<Prompt ID>.md`，不得创建、声明、修改或要求新增 Rust src/test 输出。
- Supplemental 需要影响代码时，只能写入 merge instruction，并归并到对应 Primary Prompt。
- Primary Rust module 与 src/test 文件名不得来自源文档路径、旧中间文档名、版本流水、截断尾缀或 hash 片段。
- 严格校验必须覆盖 supplemental 正文语义、source-path-like module 名称、短 hash 残片、module/output 一致性、batch 覆盖和 manifest 自洽。
