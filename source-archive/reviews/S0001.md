## 5. 严格验收下发现的问题

### P0 / 阻断项

无必然阻断项，前提是目标仓库同时拥有原 V6 Codex strict governance 文档树。

### P1 / 应修复项

1. **输出包不是完全自包含的 Codex 施工资料包。**
   阶段 START_PROMPT 多次要求读取 `docs/codex/**`、execution batch、per-file prompts，但本输出包只包含索引和映射，不包含这些原始文档实体。若用户把本包单独交给 Codex，Codex 无法读取全部被引用文件。

2. **阶段测试数据偏摘要化。**
   每阶段 `TEST_DATA.md` 多数是“根测试数据引用 + 最小 smoke case”，而不是完整的 per-stage fixture、输入动作序列、预期事件、预期错误码、snapshot diff。作为施工计划可接受；作为严格工程验收数据集仍需扩充。

3. **CI/CD 是 Markdown 内嵌 YAML，不是实际 workflow 文件。**
   这符合“md 文档组”的输出形式，但落地到仓库时还需要 Codex 抽取为 `.github/workflows/*.yml`。严格执行前建议补一个“YAML 提取与落库提示词”。

### P2 / 建议改进项

1. `test-data/rag_snapshot_cases.md` 含 `sha256-placeholder`，作为示意可接受；严格快照测试应替换为真实 hash 或明确 `example_hash` 字段。
2. S13 只要求“17 条 V1 完成标准逐项有证据”，但没有单独生成 `V1_ACCEPTANCE_EVIDENCE_MATRIX.md`。建议补一个逐条映射：验收项 → 阶段 → 测试命令 → fixture → 证据文件。
3. `manifests/README.md` 内容较短，只说明目录用途；不影响使用，但严格文档完整性可以补充 checksum 与验证命令。
4. CI 缺少 branch protection、required status checks、secrets/env matrix、artifact retention 的建议配置文档；不阻断，但可增强工程落地性。
