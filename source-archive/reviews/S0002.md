## 5. 严格验收未通过项

### P1-1：旧 v1 manifest 留在当前 `manifests/` 目录，且内容与 v2 包不一致

文件：

```text
manifests/OUTPUT_PACKAGE_MANIFEST.md
```

问题：该文件仍声明：

```text
Files: 132
```

但 v2 ZIP 实际包含 1488 个 Markdown 文件。抽样校验前 50 条 manifest 记录，发现至少 21 条 size/hash 与当前 v2 文件不一致。例如：

```text
00_INPUT_ANALYSIS_AND_TRACEABILITY.md 当前 size = 3192，旧 manifest = 5359
README.md 当前 size = 3956，旧 manifest = 7575
```

影响：Codex 可能把这个旧 manifest 当成当前输出包 manifest，从而产生错误追踪、错误 hash、错误文件总数判断。这与“清理过时信息”和“当前权威目录纯净”冲突。

建议：将该文件移至：

```text
source-archive/previous-plan-v1/manifests/OUTPUT_PACKAGE_MANIFEST.md
```

或者改名为：

```text
source-archive/previous-plan-v1/OUTPUT_PACKAGE_MANIFEST_SUPERSEDED.md
```

当前 `manifests/` 目录只保留 v2 manifest。

### P1-2：active prompt 中仍有大量 V4/V5/V6 命名，部分还作为建议 module/output 出现

复核结果：

```text
active docs/codex 中 V4/V5/V6/v4/v5/v6 命中：19030 处
active docs/codex 文件名含 v4/v5/v6：144 个
```

这些不全是错误，因为 V6 source material 本身需要 provenance；但 sampled active prompt 中存在如下语义风险：

```text
稳定 Rust module：docs_governance::v5_delivery_report
建议输出文件：docs/codex/00-index/v5_delivery_report.md
```

同一文件又声明 module 名不得携带 V4/V5 版本片段，形成内部冲突。v2 的前置清理标记虽然说明“V4/V5 仅表示 provenance”，但正文仍给出可执行的 stable module / suggested output，会干扰 Codex 的工程命名决策。

影响：这会削弱“完全由 Codex 施工”的稳定性，尤其在 documentation-or-traceability prompt 被批量执行时，可能继续生成带 `v5` / `v4_strict` 的当前目录文件或 module 名。

建议：生成一个规范化 overlay 索引，不必删除原 prompt，但必须把 active execution 使用的 `module`、`suggested_output`、`source_title` 统一映射到 current-safe 名称。例如：

```text
docs_governance::v5_delivery_report -> docs_governance::previous_delivery_report_provenance
canonical_document_boundary_v4_strict -> canonical_document_boundary_provenance
v5_delivery_report.md -> previous_delivery_report_provenance.md
```

并让 batch 执行读取该 overlay 作为最高 prompt normalization table。

### P1-3：过时信息清理方式偏“声明隔离”，不是“内容规范化”

v2 对 `docs/codex/**` 增加了 `[v2.15 自包含来源清理标记]`，这比 v1 好；但用户要求是“清理文件内容中残留的过时信息并调整文件内容格式”。严格解释下，单纯在文件头声明“旧版本词仅作 provenance”还不够，因为正文仍包含可执行的旧版本命名建议。

建议：保留原文作为 provenance，但 active 施工入口应使用以下两层：

```text
docs/codex/**                         # 原 prompt + 清理标记，作为参考/provenance
codex-active-normalized/**             # 当前施工用规范化 prompt/batch/module/output map
```

或者在 `docs/codex/00-index/` 下新增：

```text
CURRENT_NORMALIZED_PROMPT_EXECUTION_MAP.md
CURRENT_SAFE_MODULE_AND_OUTPUT_MAP.md
LEGACY_TOKEN_REWRITE_TABLE.md
```

并让 `AGENTS.md` 与所有 `START_PROMPT.md` 明确：Codex 执行 batch 前必须先应用该规范化映射。

## 6. P2 问题

### P2-1：`SELF_CONTAINED_PACKAGE_MANIFEST.md` 的自引用行 size 不准确

文件：

```text
manifests/SELF_CONTAINED_PACKAGE_MANIFEST.md
```

自引用行写成：

```text
| `manifests/SELF_CONTAINED_PACKAGE_MANIFEST.md` | 55 | `SELF-REFERENTIAL` |
```

但实际文件大小约为 399703 bytes。hash 用 `SELF-REFERENTIAL` 可以接受，size 也应标为 `SELF-REFERENTIAL` 或真实最终 size。当前写法容易让严格 manifest 校验脚本误报。

### P2-2：`fixtures` 统计存在口径不一致

`00_INPUT_ANALYSIS_AND_TRACEABILITY.md` 写：

```text
fixtures/**：34 个 fixture Markdown
```

而 ZIP 内 `fixtures/**` 实际为 35 个 Markdown 文件，其中 34 个数据 fixture + `README.md`。这不是功能性错误，但严格报告应统一口径：

```text
fixture data files = 34
fixture markdown files including README = 35
```

### P2-3：旧 hash 修复说明语义不清

`README.md` 与 `STRICT_REPAIR_CHANGELOG.md` 中出现类似：

```text
已新增 fixtures/** 扩展测试数据，并修复 <hash>
```

该 hash 原本是旧 placeholder token 的替换目标，但上下文不充分。建议写成：

```text
已移除旧 placeholder hash token，并替换为确定性测试 hash：...
```

### P2-4：仍有 1 个 Markdown 文件存在行尾空格

文件：

```text
reviews/PREVIOUS_STRICT_ACCEPTANCE_REVIEW.md
```

问题很小，但严格格式化验收应清理。

### P2-5：fixture 仍以“验收输入”居多，尚非完整自动化断言全集

v2 fixture 已经比 v1 强很多，但部分 stage fixture 主要检查 required documents / required evidence，缺少更细颗粒的 expected GameEvent、DecisionRecord、DiceRoll、projection hash、export diff。`PER_STAGE_FIXTURE_EXPANSION_PLAN.md` 已要求 Codex 后续扩展，这适合作为施工方案；但若验收标准要求“可直接生成全量自动化测试”，仍需 v2.1 继续补齐。
